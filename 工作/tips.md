# 基础知识点
## C/C++
### Static关键字

1. 用在函数内部修饰局部变量，改变存储周期，从自动存储期改为静态存储期，程序开始运行时初始化一次，调用结束之后，该变量也会保存。如果不显式初始化，则编译器会初始化为0
2. 在函数外部，修饰全局变量和函数，将全局变量或者函数的作用域定义在源文件中，其他源文件无法使用`extern`来引用


### volatile 关键字

> 告诉编译器：这个变量的值可能会在程序自身之外被意外改变，确保每次访问变量时都直接从其内存地址中读取，而不是使用寄存器中的缓存值

### typedef

> typedef 定义的新名字通常放在声明语句的末尾，与普通变量声明的位置类似，但它是类型名而不是变量名。
> 
> typedef 是在编译阶段处理的，而 #define 是预处理器进行的简单文本替换。typedef 受作用域规则限制
>
> 使用 typedef 可以增强代码的可移植性,例如在跨平台开发中，为不同平台下大小不同的整数类型定义统一的别名（如 int32_t）。

```c
typedef 原类型名 新别名;
//为变量取别名
typedef unsigned int uint;      // 给 unsigned int 起别名 uint
typedef long long int64;        // 给 long long 起别名 int64
//为结构体取别名
// 方式一：先定义结构体，再用 typedef 起别名
struct Point {
    int x;
    int y;
};
typedef struct Point Point;      // 以后 Point 就等价于 struct Point

// 方式二：定义结构体的同时使用 typedef
typedef struct {
    int x;
    int y;
} Point;                         // 这里 Point 是结构体的别名，可以省略结构体标签

//为指针类型取别名
typedef int* IntPtr;             // IntPtr 是指向 int 的指针类型
int a = 10;
IntPtr p = &a;                // 相当于 int* p = &a;

//为函数指针取别名
// 定义一个函数类型，该函数接收两个 int 参数，返回 int
typedef int (*BinaryOp)(int, int);
BinaryOp op;                  // 声明一个函数指针变量 op
op = add;
op(5,3);
```

### const 关键字

用于定义常量，表示“只读”

1. 指向常量的指针（指针指向的内容不可变）
   ```c
    const int *p; 
    // *p = 10;    // 错误！不能通过 p 修改它指向的数据
    p = &some_var; // 正确，p 本身可以指向别的地址
    ```
2. 常量指针（指针本身不可变，即指向的地址不可变）
    ```c
    int *const p = &var;
    *p = 20;       // 正确，可以修改指向的数据
    // p = &other; // 错误！p 的指向不能改变
    ```
3. 指向常量的常量指针（两者都不可变）
    ```c
    const int *const p = &var;
    // *p = 20;    // 错误
    // p = &other; // 错误
    ```
4. const 修饰指针传参的意义：防止意外修改原数据

### extern关键字

`extern`关键字用于声明变量/函数
```c
extern int global_var; //只是声明（不能初始化），而 int global_var; 是定义（分配内存）
```
1. 跨文件访问全局变量/函数
在一个源文件中定义全局变量，其他源文件通过 `extern` 声明后即可使用，实现数据共享。
通常在头文件中用 `extern` 声明变量，在对应的源文件中定义变量，这样包含头文件的任何源文件都可以访问该变量。

2. `extern "C"`这是 C++ 中的用法，用于告诉 C++ 编译器以 C 的方式链接函数，用于在 C++ 中调用 C 编写的库。

### sizeof和strlen
1. `sizeof(arr)`表示的是数组的大小，`sizeof(ptr)`表示的是指针的大小(4字节或8字节)
2. `sizeof(字符串)`含有`\0`，`strlen(字符串)`不含有`\0`

### 常见位操作

> 位掩码：使用左移<<，右移>>，1<<7生成第7位的掩码0x10000000，1>>0生成第0位的掩码 0x00000001


1. 提取特定位
```c
//获取data的第3位
uint8_t data = 0b10101010;
uint8_t result = data & (1 << 3);   //result = 1;
```
2. 设置特定位
```c
//设置data的第4位为1
data  |=  (1 << 4);
//设置data的第3位和第5位为1（不连续位）
uint8_t mask = (1 << 3) | (1 << 5); //生成第3、5位的掩码
data |= mask;
```

3. 切换特定位
```c
//反转data的第2位
data  ^= (1 << 2)

//反转data的第3位和第5位
uint8_t mask = (1 << 3) | (1 << 5); //生成第3、5位的掩码
data ^= mask;
```
4. 清除特定位
```c
//清除data的第2位
data &= ~(1 << 2);

//清除data的第3位和第4位
data &= ~((1 << 3) | (1 << 4));
```

### 结构体、联合体和内存对齐
> 结构体，每个成员都有自己的内存空间，总大小>=各成员大小，可以同时访问所有的成员
>
> 
>联合体，所有成员共享一块内存，总大小==最大成员的大小，只能访问最后一个赋值的成员

#### 什么是对齐？

答：编译器在给结构体分配内存时，按照某种规则在成员之间插入填充字节，使得每个成员落在自然对齐的边界上（数据的起始地址必须是该数据类型大小的整数倍。）

#### 为什么要对齐？

某些CPU访问未对齐的数据会出错；对齐的数据访问起来更快

#### 常见的对齐方式有哪些？

- 自然对齐，指数据类型按照其自身的对齐字节数存储，比如`char`按照1字节对齐，`int` 按照4字节对齐
- 编译器指令对齐，`#pragma pack(n)`可以显式控制按照1、2、4、8字节对齐

#### __attribute__((aligned(2n)))和`#pragma pack(2n)`的区别?

`__attribute__((aligned(2n)))`主要影响结构体整体的对齐，不影响结构体内部成员的对齐方式。成员仍然按照各自的自然对齐方式，

`#pragma pack(2n)`强制结构体的每个成员按照4字节对齐，忽略成员的自然对齐需求。

### 条件编译 (预处理器指令，在编译之前执行)
```c
#ifdef MACRO <=> #if defined(MACRO)
#ifndef MACRO <=> #if !defined(MACRO)
```
#### 条件编译在什么情况下会被用到
1. 防止头文件被重复包含
```c
#ifndef _MYHEADER_H_  // 如果_MYHEADER_H_没有被定义
#define _MYHEADER_H_  // 则定义它，并编译以下内容

// ... 头文件的真实内容（函数声明、宏定义等）...

#endif /* _MYHEADER_H_ */
```

2. 调试代码，开发阶段加入调试语句，通过设置宏控制开启
3. 跨平台移植，针对不同操作系统编写特定代码
4. 功能开关，为不同的客户编译不同通能的软件版本

### 函数指针和回调函数
```c
int (*pfunc)(int, int);		//函数指针声明
int (*func_array[5])(int, int);  //函数指针数组，每个元素为int (*)(int, int)类型的函数指针
```
函数名本身就是函数的地址
> 回调函数是通过函数指针调用的函数，将自己定义的一个函数（回调函数）的指针作为参数传递给另一个函数（主调函数），主调函数在某个事件发生时反过来调用自己定义的函数。

#### 回调的简单例子

```c
#include <stdio.h>

// 定义回调函数类型：接受一个int参数，无返回值
typedef void (*Callback)(int);

// 模拟一个耗时操作，完成后调用回调
void doWork(int data, Callback cb) {
    printf("Processing data: %d\n", data);
    // 模拟一些工作
    int result = data * 2;
    // 工作完成，调用回调，传递结果
    cb(result);
}

// 定义一个回调函数，用于处理结果
void myCallback(int result) {
    printf("Callback called with result: %d\n", result);
}

int main() {
    int input = 10;
    // 将myCallback作为回调传递给doWork
    doWork(input, myCallback);
    return 0;
}
```

#### 回调函数的应用
- 异步编程：网络编程，数据到达后，操作系统调用定义的回调
- 事件驱动：如在`http`服务端为不同的请求注册不同的路由，调用回调函数进行处理

#### 为什么不直接调用，而要用回调？
- 直接调用——主调控制一切，高耦合，代码复用性低，可扩展性低；
- 回调——主调只需要定义流程和时机，具体做什么通过回调交给外部实现，实现了控制反转，松耦合，代码复用性高，可扩展性低。

### 内联函数inline的使用场景和注意点？

答：主要针对高频使用的小函数，内联可以减小开销，通常定义在头文件中.
inline 仅是建议，编译器可能忽略。内联会在每个调用点复制函数体，若函数较长或调用频繁，会导致可执行文件体积剧增。类内部定义的函数，也属于内联函数

#### 内联函数和宏函数的区别？
- 宏函数：通过 `#define` 预处理器指令定义的带参数的宏，在预处理阶段进行简单的文本替换（预处理阶段）作用域全局
- 使用 `inline` 关键字修饰的函数，建议编译器在调用处将函数体展开，避免函数调用开销。实际是否内联由编译器决定（编译阶段）作用域与一般函数一样，支持函数重载

### 野指针的产生以及如何避免？

产生原因
1. 指针未初始化
2. 释放后未置空
3. 函数返回局部变量地址
4. 重复释放

针对上面提到的点进行避免

### calloc/malloc/realloc的区别？
它们都用于在堆上分配内存

1. `void *malloc(size_t, size);`功能：分配一块连续的内存区域,大小为 `ssize` 字节。不初始化
2. `void *calloc(size_t nmemb, size_t, size);`功能：分配一块连续的内存区域，足以容纳 `nmemb` 个大小为 `size` 的元素，即总大小为 `nmemb * size` 字节。初始化：自动将每一位初始化为 0（即所有字节置零）
3. `void *realloc(void *ptr, size_t new_size);`功能：调整之前分配的内存块的大小（扩大或缩小）。如果 `ptr` 为 `NULL`，则行为等同于 `malloc(new_size)`。如果 `new_size` 为 0，且 `ptr` 非 `NULL`，则行为等同于 `free(ptr)`，并返回 `NULL`。`ptr` 为之前由 `malloc`、`calloc` 或 `realloc` 返回的指针，`new_size` 为新的大小（字节）。数据保留：如果新大小大于原大小，原数据保持不变，新增部分的内容未初始化（类似 malloc）；如果新大小小于原大小，原数据会被截断，保留前 `new_size` 字节。
4. 所有由这三个函数分配的内存都需用 free 释放

### 大结构体传参时为什么使用指针或者引用？
答：避免拷贝开销（效率），节省栈空间，防止栈溢出

### C++的引用和指针有什么区别？为什么引入？

答：指针和引用都用于间接访问数据，引用在声明的时候就必须初始化且不能更换目标，相当于变量的一个别名，指针则可以声明之后再进行赋值；指针可以进行算术运算，引用不支持。

### 为什么引用比指针更安全？
- 由于初始化特性，且引用不能绑定空值，指针可能为null，因此引用比指针更安全。

- 指针指向的对象被释放后，指针仍然保留地址，使用可能会导致未定义行为，而引用的是与其被绑定的对象生命周期同步的。

- 引用是终身绑定的，指针可以随意更改，因此可能会导致一些误操作。

```c
#ifdef __cplusplus
```

### 不使用库函数使用字符串操作
```c
//Strlen
size_t my_strlen(const char *str) {
    const char *p = str;          // 1. 用一个指针 p 指向字符串开头
    while (*p != '\0') p++;        // 2. 遍历直到遇到字符串结束符 '\0'
    return p - str;                // 3. 指针相减得到字符个数（长度）
}

//Strcpy
char *my_strcpy(char *dest, const char *src) {
    // 1. 检查指针有效性
    if (dest == NULL || src == NULL) {
        return NULL;
    }

    // 2. 保存目标字符串的起始地址，用于返回
    char *ret = dest;

    // 3. 逐个字符复制，包括 '\0'
    while ((*dest++ = *src++) != '\0') {
        // 空循环体，所有操作都在条件判断中完成
    }

    return ret;
}
//Strcat
char *my_strcat(char *dest, const char *src) {
    // 1. 检查指针有效性
    if (dest == NULL || src == NULL) {
        return NULL;
    }

    // 2. 找到 dest 的末尾（即 '\0' 的位置）
    char *d = dest;
    while (*d != '\0') {
        d++;
    }

    // 3. 从 dest 的末尾开始，执行 strcpy 操作
    while ((*d++ = *src++) != '\0') {
        // 空循环体
    }

    return dest;
}
//Strcmp
int my_strcmp(const char *s1, const char *s2) {
    // 1. 检查指针有效性（标准库不检查，但自己实现时最好检查）
    if (s1 == NULL || s2 == NULL) {
        // 处理错误，可以返回一个特定值，比如认为NULL的字符串最小
        if (s1 == NULL && s2 == NULL) return 0;
        if (s1 == NULL) return -1;
        return 1;
    }

    // 2. 逐个字符比较
    while (*s1 != '\0' && *s1 == *s2) {
        s1++;
        s2++;
    }

    // 3. 返回值（直接相减符合返回值的定义）
    return *(const unsigned char*)s1 - *(const unsigned char*)s2;
}
```

### memcpy 和 memmove

答：这两个函数都能实现内存的复制，但是最根本的区别是，`memcpy` 不会处理内存不重叠的情况，`memmove`会检查并处理内存重叠情况，比如，当目标地址在源地址之后时，`memcpy`顺序复制会覆盖尚未复制的源数据，导致错误，这时使用`memmove`函数就会更安全。(结合项目中的交换图片顺序)

### 进程和线程的核心区别？
答：
- 资源占用：进程大，线程小。进程是资源分配的最小单位，线程是CPU调度的最小单位，进程拥有独立的地址空间（堆栈，数据，代码），进程间资源隔离。线程与其他线程共享地址空间，只拥有独立的栈和寄存器上下文
- 调度开销：进程大，线程小。进程切换涉及虚拟地址空间的切换，线程切换仅需保存/恢复寄存器。
- 通信方式：进程复杂，需要借助内核，管道，消息队列，共享内存，socket等，线程简单，可以直接读写全局变量，但需要互斥锁等防止竞争

### 什么时候用进程，什么时候用线程？
答：嵌入式系统分成裸机（无OS），RTOS和Linux三大类
- 系统复杂，需要模块隔离，防止相互干扰，稳定，选择进程
- 数据量大，交互频繁，追求低延迟，实时性要求	选择线程
- 在资源受限的场景下，选择线程，开销更小
- 核心业务解耦，选择进程，数据采集，处理联动，选择线程

### 死锁的四个必要条件和规避方案
答：四个必要条件：
- 互斥（资源只能被一个进程或线程占用）
- 持有并等待（执行单元已经占用了至少一个资源，同时又在等待获取其他被占用的资源）
- 不可以剥夺（已获得的资源在未使用完之前，不能被其他执行单元强行剥夺，只能由持有者主动释放）
- 循环等待条件，存在一组等待进程 {P1, P2, ..., Pn}，P1 等待 P2 占有的资源，P2 等待 P3 占有的资源，...，Pn 等待 P1 占有的资源，形成一个闭环

规避方案：
- 固定锁顺序，所有线程按照相同的顺序获取锁，避免循环等待
- 使用定时锁：若超时则回退并释放已有锁，避免无限等待。
- 增加死锁检测日志
- 减小锁持有范围

### 进程间通信
答：常见方式有管道、共享内存、消息队列，信号量，本地套接字
- 共享内存：多个进程映射同一块物理内存，通过直接读写该内存通信，最快适合大数据传输，如视频码流等，但是需要配合信号量或者互斥锁做同步
- 本地套接字通用性最强，适合跨进程的复杂通信，默认双向，缺点，需要手动管理连接，数据仍需在内核与用户空间之间拷贝。
- 消息队列，消息以链表形式存放在内核，每个消息有类型和长度，自带同步，无需额外锁，不适合大量数据，会有两次拷贝。用户→内核，内核→用户
- 管道，内核维护一个环形缓冲区，一端写入，另一端读出。适合简单的父子进程通信，数据只能单向流动，自带同步。

### linux文件IO


|特点|系统调用(open/read/write)|C标准库函数(fopen/fread/fwrite)|
|-|-|-|
|所属层次|内核态，直接与操作系统进行交互|运行在用户态|
|操作对象|文件描述符 `int fd`|文件指针 `FILE *fp`|
|缓冲机制|无用户态缓冲，每次调用直接发起内核 I/O|有用户态缓冲，数据先存入用户空间缓冲区|
|调用开销|每次调用涉及到内核态和用户态的切换，开销大(涉及到现场的保护恢复和状态的切换)|多数时候在用户态操作缓冲区，减少了系统调用次数（`fwrite`只有在缓冲区满的时候，才会调用`write`系统调用）|

内核态缓冲区会有一个页缓存（内存区域）的概念，通常内核`write`不会立刻写入磁盘，而是先写入页缓存，后续再异步刷入磁盘，当`read`时，会先检查页缓存，如果直接命中则直接从内存返回，减少磁盘 IO

标准库性能好，但是系统调用实时性更好。非阻塞 I/O：系统调用支持 O_NONBLOCK 标志，而标准库函数默认阻塞。


### 内存映射原理
答：mmap 是一种将文件或设备映射到进程地址空间的方法，使得应用程序可以像访问内存一样直接读写文件，而无需通过传统的 read/write 系统调用。mmap 会在进程的虚拟地址空间中创建一段映射区域，将该区域与某个文件（或匿名内存）的物理页关联起来。 mmap 创建映射后，返回一个指向映射区域的指针，后续操作就像操作普通内存一样。

 内存映射就是将文件或者设备直接映射到进程的虚拟地址空间，使得文件操作像操作内存，内存映射是进程间通信的一种重要实现方式，适用于大文件读写，进程间共享内存就是通过内存映射实现的。消除了数据拷贝，提升性能


### mmap 相比 read/write 有什么优势？
减少数据拷贝，适合大文件；简化随机访问；支持进程间共享。



### 编译过程，预处理、编译、汇编、链接四个阶段分别做什么？
答：预处理：处理 #include、宏定义、条件编译等，生成 .i 文件。

编译：将预处理后的文件进行词法、语法、语义分析，优化后生成汇编代码 .s。

汇编：将汇编代码转换为机器指令，生成可重定位的目标文件 .o。（静态库是在这一步生成的，由编译工具链中的ar生成）

链接：将多个目标文件和库合并，进行符号解析（将所有符号定义和引用进行关联）、地址分配（为各个段分配虚拟内存地址）和重定位（根据分配的地址，修正引用处的地址值），最终生成可执行文件（如 ELF）（动态库是在这一步生成的，由编译工具链中的ld链接器生成）

### 静态库和动态库的区别是什么？
答：静态库和动态库最核心的区别在于链接的时机。静态库在编译链接阶段被整合进可执行文件，而动态库则在程序运行时才被加载。
静态库，代码被复制到可执行文件中，生成的文件体积大
动态库，可执行文件只保留引用，体积小

### 符号冲突——强符号和弱符号
答：
- 强符号：通常指函数定义、已初始化的全局变量。
每个强符号在链接时只能出现一次，否则会引发“多重定义”错误。

- 弱符号主要包括：未初始化的全局变量。显式使用 __attribute__((weak)) 修饰的全局变量或函数。允许多次定义，链接器会在强符号和弱符号之间按规则选择。

链接规则：
- 强符号不允许重复
- 强符号覆盖弱符号
- 多个弱符号，链接器可能会选择某一个（取决于链接器），实际在定义变量和函数的过程中，应该避免这种情况。

### 定时器
答：定时器启动后，计数器按时钟节拍不断变化（递增或递减）。当计数值达到预设的阈值时，硬件自动触发事件（如中断、电平翻转），同时计数器可清零并重新开始，形成周期性定时。定时器分成软件定时器和硬件定时器，软件定时器中，timeerfd是嵌入式首选，将定时器转为fd，配合epoll可以实现大量定时器。

### 看门狗

### TCP握手挥手过程，状态转换，CLOSE_WAIT，TIME_WAIT
答：三次握手
1. 客户端发送 SYN 报文（同步序列号），进入 SYN_SENT 状态。
2. 服务端收到后，回复 SYN+ACK 报文，进入 SYN_RCVD 状态。
3. 客户端收到后，发送 ACK 报文，双方进入 ESTABLISHED 状态。
四次挥手
1. 主动关闭方（客户端）调用 close()，发送 FIN，进入 FIN_WAIT_1。
2. 被动关闭方（服务端）收到 FIN，回复 ACK，进入 CLOSE_WAIT 状态；主动方收到 ACK 后进入 FIN_WAIT_2。
3. 被动关闭方也调用 close()，发送 FIN，进入 LAST_ACK 状态。
4. 主动关闭方收到 FIN，回复 ACK，进入 TIME_WAIT，等待 2MSL（最大报文段生存时间）后进入 CLOSED；被动方收到 ACK 后进入 CLOSED

- CLOSE_WAIT:表示本地应用程序尚未调用 close() 关闭连接，但对方已经关闭了写通道，CLOSE_WAIT 堆积：被动关闭方未调用 close，通常为程序 bug
> TIME_WAIT:主动关闭方（发送最后一个 ACK 后进入）,在高并发服务器上，大量 TIME_WAIT 会占用系统资源（端口、内存），通常可调整内核参数来优化。TIME_WAIT 过多：高并发短连接场景正常，但需调整系统参数（如 tcp_tw_reuse）以优化资源。


可以通过netstat查看链接状态

### socket

|接口|调用方|状态变化|
|-|-|-|
|`bind`|服务端|无状态变化，仅绑定套接字地址|
|`listen`|服务端|套接字从 `CLOSED`->`LISTEN`，内核准备接收连接|

主要使用UDP协议，Get和Set使用161端口，Trap和Inform使用162端口。

- Get/GetNext/GetBulk：NMS 向 Agent 请求一个或多个对象的值。
- Set：NMS 向 Agent 写入值，修改设备配置。
- Trap/Inform：Agent 主动向 NMS 报告异常或事件（如接口 down）。Trap 无需确认，Inform 需要确认。

- v1：最基础，使用团体字（community string）作为明文密码，无加密
- v2c：增强了错误码类型（如不再区分 noSuchName 等），但仍使用团体字明文传输，是当前最广泛部署的版本
- v3：增加了用户安全模型，提供认证（MD5/SHA）和加密（DES/AES），可配置不同的安全级别（noAuthNoPriv、authNoPriv、authPriv）。
管理端（NMS，Network Management Station）：如 SolarWinds、Zabbix，主动发起 Get/Set 请求，或接收 Trap。

- 代理端（Agent）：运行在被管理设备上（如你的智能服务器），响应 NMS 的请求，并可能主动发送 Trap

- MIB（管理信息库）：定义了被管理设备的变量集合，是树形结构。每个节点由 OID（对象标识符）唯一标识，例如 1.3.6.1.2.1.1.1 对应 sysDescr（系统描述）。

- OID（对象标识符）：由一串数字组成，如 .1.3.6.1.4.1.26864 通常是企业私有 MIB 的根节点（海康的企业 OID 可以查一下）。

### webservice
答：Web Services 使用 XML 来编解码数据，并使用 SOAP 借由开放的协议（如HTTP）来传输数据。通过使用 Web Services，可以在不同的应用程序与平台之间来交换数据。

SOAP 协议使用 XML 格式来定义消息的结构和内容。

WSDL 本质就是一种使用 XML 编写的文档。这种文档可描述某个 Web service。它可规定服务的位置，以及此服务提供的操作（或方法）

### 内存调优（内存碎片解决）
答：使用内存池（Memory Pool）
原理：预分配一大块内存，内部划分为固定大小的块。分配时直接从池中取一块，释放时归还。完全消除外部碎片，内部碎片取决于块大小选择。

- 定长内存池：内存池中所有的内存块大小是固定的，初始化时，将大块内存划分成N个固定大小的块，用一个空闲链表将所有的空闲块串联起来（块内有字节存储指向下一个空闲块的指针）分配时，从链表头部取下第一个空闲块，返回其地址。释放时，将块重新放回链表头部。整个过程只需链表指针操作，无系统调用，极快。如果对象实际大小小于块大小，会造成空间浪费。

不灵活：不能分配大于块大小的内存。
- 变长内存池允许分配不同大小的内存块。它仍然预申请一大块内存，但内部需要更复杂的管理算法（如伙伴算法、最佳适应算法等）来跟踪空闲区域，并尽量减小碎片，灵活，能分配任意大小（受池总大小限制），分配/释放有一定开销（可能需要遍历或合并）

### 栈溢出
答：栈溢出是指程序运行时，向栈中写入的数据超出了栈帧（stack frame）所分配的空间，从而覆盖了相邻内存区域（如返回地址、局部变量、函数参数等），导致程序行为异常甚至崩溃。
常见原因：
- 递归过深	未正确设置递归终止条件，或递归层数太多	每次递归都消耗栈空间，最终耗尽
- 局部数组过大	在函数内定义 char buf[1024*1024]

### 嵌入式Linux和RTOS对比
答：
- 实时性，嵌入式linux任务响应时间不确定，毫秒级（软实时），RTOS任务响应时间确定，微秒级（硬实时），Linux场景：智能家居中控，智能交通，RTOS场景：工业电机控制，汽车电子等场景，需要及时响应

## 视频编码相关

从采集视频开始，经过编码（H.264/H.265），封装成MP4或PS流，然后通过RTMP或RTP传输，客户端通过RTSP控制播放等。

> 视频编码: 又称视频压缩，就是将原始视频数据压缩，减少体积，方便存储和传输。

### 一些基本术语

#### 分辨率

> 定义：视频画面的像素数量，例如 1920x1080（1080p）、3840x2160（4K）

分辨率决定画面的清晰度和细节。

分辨率类似于一张画布的尺寸，分辨率越高，画布越大，能够画的细节越多。

分辨率高 ≠ 一定清晰！如果码率不足，高分辨率反而会模糊（就像大画布上颜料不够，只能涂鸦）。

常见标准：720p（高清）、1080p（全高清）、4K（超高清）。

#### 帧率（FPS）

> 每秒显示的帧数，例如 60FPS（游戏/体育）

决定了视频的流畅度，尤其是快速运动的画面。类似于以前的翻页漫画，翻页越快，帧率越高，动画越流畅。


高帧率需要高码率支持，否则画面就会模糊或者卡顿。

#### 码率

> 每秒传输的数据量，单位是 Mbps 或 Kbps

决定视频的压缩质量和文件大小。如果分辨率类比于画布的话，那么码率就类比于给画布分配的“颜料”，码率越高，颜料越充足，画质越细腻。

恒定码率（CBR）：码率固定，适合稳定带宽（如直播）。

可变码率（VBR）：码率动态调整，复杂画面分配更多码率（如电影）。

#### 几个经典问题

1、为什么有的4K视频很模糊？

可能码率太低，无法支撑高分辨率（“大画布颜料不足”）。

2、 为什么高帧率视频卡顿？

可能码率不足，或设备性能不够（解码速度跟不上高帧率）。

3、如何平衡三者？

- 优先保证码率足够：码率是画质的核心。
- 分辨率根据显示设备选择：手机看1080p足够，没必要强行4K。
- 帧率按内容需求：静态画面（如讲座）24FPS足够，动态画面（如游戏）需60FPS。



#### 常见的视频编码格式

|常见视频压缩格式|说明|应用|
|-|-|-|
|H.264（AVC）|最常见的编码标准，压缩效率高，兼容性好，广泛用于视频网站、监控等。|流媒体、监控、视频会议|
|H.265（HEVC）|H.264 的升级版，压缩效率提升约50%，但计算复杂度更高，适合 4K/8K 等高分辨率视频。|4K流媒体、超高清广播|
|AV1|压缩效率接近H.265，优于VP9。编码速度较慢，解码复杂度高（需硬件支持）|开源流媒体平台（如YouTube、Netflix逐步支持）、WebRTC视频通话|
|VP9|谷歌推出的开源编码标准，压缩效率优于H.264，但弱于H.265。|主要支持Chrome、YouTube等谷歌生态。|
|H.266|压缩效率比H.265再提升30-50%，支持8K/VR/360°视频|计算复杂度极高，目前仅专业领域使用|


#### 编码时生成的关键帧

|关键帧类型|说明|
|-|-|
|I帧（关键帧）|完整画面，不依赖其他帧，解码起点。|
|P帧（预测帧）|基于前一帧（I或P帧）的差异压缩，体积更小。|
|B帧（双向预测帧）|基于前后帧的差异压缩，压缩率更高，但延迟增加（如视频会议不用B帧）。|


#### 视频封装格式

> 视频封装: 将编码后的视频、音频、字幕等打包成一个文件，或用于传输流。

|常见封装格式|说明|应用|
|-|-|-|
|MP4|最常见的封装格式。<li>支持 H.264/H.265/AV1 等视频编码，以及 MP3 等音频编码，适合本地存储和在线点播（如视频网站）。<li>文件体积小，兼容性极强（手机、网页、播放器全支持）<li>不支持多音轨或多字幕（通常只能包含1条视频+1条音频+1条字幕）|在线视频（如YouTube、B站）、本地存储、移动设备播放|
|MKV|<li>高度灵活的开源格式，支持几乎所有编码格式（H.264、VP9、FLAC等）。<li>支持多音轨、多字幕、章节、菜单等复杂功能。<li>适合高质量内容封装（如蓝光原盘备份）。<li>兼容性较差（部分老旧设备或播放器无法直接播放）。|高清影视资源、多语言字幕/音轨的电影、DIY视频制作|
|PS流（Program Stream）|用于 DVD 等存储场景，<li>封装较简单，对传输错误不敏感。网络传输中易因丢包导致数据损坏。<li>不支持现代编码格式（如H.265）和多路复用复杂流|DVD和早期蓝光光盘，监控录像存储|
|TS流（Transport Stream）|<li>用于广播电视和流媒体（如直播），支持分片传输，抗干扰性强。<li>专为流媒体传输设计，支持分片传输和容错（抗网络丢包）。<li>文件结构复杂，体积较大。|直播流（如HLS协议中的.ts分片）、数字电视（DTV）、蓝光碟片|
|AVI|微软推出的老旧格式<li>兼容性较好。支持多种编码（如DivX、Xvid），但扩展性差。<li>不支持现代功能（如H.265、章节、流媒体传输）。<li>文件体积大（未优化压缩）。|早期Windows系统视频、部分监控录像|
|FLV|<li>早期用于Adobe Flash播放器的流媒体格式。<li>支持H.264编码，结构简单，适合低延迟传输。<li>已被淘汰（Flash技术停用），但仍有部分直播推流使用|旧版网页视频、RTMP直播推流（如早期斗鱼直播）|
|MOV|<li>苹果公司开发，支持高画质和多轨道（视频、音频、文本）。<li>兼容性较好（需QuickTime或支持MOV的播放器）。<li>文件结构复杂，体积较大。|专业视频剪辑（Final Cut Pro）、苹果生态内的视频存储|
|WebM|<li>谷歌推出的开源格式，专为网页设计。<li>支持VP9、AV1视频编码和Opus音频编码。<li>兼容性依赖浏览器（Chrome、Firefox支持较好）|HTML5网页视频（如YouTube的WebM格式）、低带宽流媒体|

#### 流媒体传输协议

> 流媒体传输协议：在网络上实时传输音视频数据，支持直播、点播等场景。

|传输协议|功能|应用|
|-|-|-|
|RTSP（Real-Time Streaming Protocol）|控制媒体流的播放、暂停等（类似“遥控器”），不直接传输数据。|监控摄像头（如通过VLC播放RTSP流）|
|RTP（Real-Time Transport Protocol）|实际传输音视频数据，常与RTSP配合使用（RTSP控制，RTP传输）。|支持时间戳和序列号，解决网络抖动和乱序问题。|
|RTMP（Real-Time Messaging Protocol）|基于TCP的协议，低延迟（1-3秒），但受限于TCP可能在高丢包率下卡顿。|最初用于Flash直播，现仍用于推流（如主播推流到服务器）。|
|HLS（HTTP Live Streaming）|苹果推出的基于HTTP的流媒体协议，将视频切分为TS小文件，适合自适应码率（如B站、YouTube）。||


#### 典型流程

下面介绍几个典型场景

- 直播场景：
主播端 → 摄像头采集 → H.264编码 → 封装为FLV → RTMP推流到服务器 → 服务器转HLS/RTMP → 观众通过HTTP/HLS或RTMP播放。

- 监控场景：
摄像头 → H.265编码 → 封装为PS流 → RTSP控制 + RTP传输 → 客户端接收并解码播放。

- 点播网站：
视频文件 → H.264编码 → 封装为MP4 → 通过HTTP渐进下载或HLS分片传输 → 浏览器播放。

#### 音视频编解码流程

> 音视频编解码本质是一种压缩与解压缩的技术。未经压缩的原始音频数据量巨大，不便于存储和网络传输。

以 H264/ACC 编码为例，视频编码的核心思想是对 I 帧进行类似图片的压缩，针对 P 帧，只记录与前一帧的变化。（编码的核心是去除冗余）。编码过后的视频和音频需要按照一定的格式（如FLV，MP4）组合在一起，并加上时间戳，实现音画同步。之后再通过传输协议 RTSP/RTMP 推流到服务器，流媒体服务器接收到之后分发，播放器从服务器拉流，再进行解码

解码的过程是编码的逆过程。
